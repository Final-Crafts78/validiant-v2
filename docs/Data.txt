# Projects Command Center — Implementation Plan

## Goal

Transform the **Projects page** into the **central command center** of the Validiant platform. Every task-related workflow — creation, assignment, bulk operations, status tracking, column customization, filtering, and analytics — should live **inside Projects**. Field executives see tasks from multiple projects on a unified dashboard, and each task's detail modal displays the originating project name.

This plan also identifies **12+ existing features that have been built but not wired up**, and brings them all under the Projects umbrella.

---

## User Review Required

> [!IMPORTANT]
> **Schema Migration**: This plan adds a new DB table (`project_column_definitions`) and a new column (`assigned_date`) to the `tasks` table. This requires a Drizzle migration + `drizzle-kit push`. Please confirm you're comfortable with a schema change before we proceed.

> [!WARNING]
> **Task Page Restructuring**: The current standalone `/tasks` page will be demoted to a redirect/legacy route. All task management will move inside `/projects/[id]/tasks`. The old `/tasks` page can optionally redirect to the active project. Please confirm this navigation change is acceptable.

> [!IMPORTANT]
> **Customizable Columns**: The plan stores column definitions in JSONB on `project.settings` (lightweight) AND in a dedicated `project_column_definitions` table (full-featured). This gives you both speed and power. Please confirm which approach you prefer, or if you'd like **both** (JSONB for fast reads, table for audit trail).

## Proposed Changes

### Component 1: Database Schema Extensions

#### [MODIFY] [schema.ts](file:///c:/Users/manso/OneDrive/Documents/Validiant-v2/validiant-v2/apps/api/src/db/schema.ts)

Add to the `tasks` table:
- `assigned_date` (timestamp) — set at the moment a task is assigned, per Data.txt spec  
- `notes` (text) — admin instructions field per Data.txt spec

Add a new table `project_column_definitions` for per-project customizable columns:

```sql
project_column_definitions (
  id           UUID PRIMARY KEY,
  project_id   UUID NOT NULL REFERENCES projects(id) ON DELETE CASCADE,
  key          VARCHAR(100) NOT NULL,      -- machine key, e.g. 'client_name'
  label        VARCHAR(200) NOT NULL,      -- display label, e.g. 'Client Name'
  type         VARCHAR(50) NOT NULL,       -- 'text' | 'number' | 'date' | 'select' | 'url' | 'pincode' | 'map' | 'user'
  position     INTEGER NOT NULL DEFAULT 0, -- column order
  is_visible   BOOLEAN NOT NULL DEFAULT true,
  is_required  BOOLEAN NOT NULL DEFAULT false,
  is_system    BOOLEAN NOT NULL DEFAULT false,  -- system columns can't be deleted
  color        VARCHAR(7),                 -- optional hex color for column header
  width        INTEGER,                    -- optional column width in px
  options      JSONB DEFAULT '[]',         -- for 'select' type: array of {value, label, color}
  default_value TEXT,                      -- optional default
  metadata     JSONB DEFAULT '{}',         -- extensible config
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(project_id, key)
)
```

**Rationale**: This gives every project a fully customizable column schema. System columns (title, status, assignee, etc.) are pre-seeded with `is_system = true` and can be renamed/recolored but not deleted. Admins can add unlimited custom columns.

## Debug Logging Strategy

Every layer will include structured debug logs with the prefix format `[Module:Action]`:

### API Layer Logs
```
[ProjectColumns:List]     → columns fetched for project {projectId}
[ProjectColumns:Create]   → column created {key, label, type, projectId}
[ProjectColumns:Update]   → column updated {colId, changes}
[ProjectColumns:Delete]   → column deleted {colId, projectId}
[ProjectColumns:Reorder]  → columns reordered {projectId, newOrder}
[TaskAssign:Execute]      → task assigned {taskId, userId, assignedDate, previousStatus, newStatus}
[TaskAssign:Preserve]     → terminal status preserved on reassignment {taskId, status}
[TaskCreate:Unassigned]   → task created in unassigned pool {taskId, projectId}
[TaskCreate:Assigned]     → task created with assignee {taskId, userId, status: 'Pending'}
[TaskFilter:Apply]        → filters applied {status, assignee, pincode, projectId}
```

### Frontend Logs
```
[ProjectDetail:Mount]     → project detail page mounted {projectId, orgSlug}
[ProjectDetail:TabChange] → tab changed {tab, projectId}
[ColumnCustomizer:Save]   → column definition saved {colId, changes}
[TasksTable:ColumnsInit]  → dynamic columns initialized from definitions {count, projectId}
[BulkAssign:Execute]      → bulk assign triggered {taskIds, assigneeId}
[BulkStatus:Execute]      → bulk status change triggered {taskIds, newStatus}
[TaskSlideOver:Open]      → task detail opened {taskId, projectName}
```

### Middleware/Store Logs
```
[WorkspaceStore:ProjectChange] → active project changed {prevId, newId}
[QueryInvalidation:Tasks]      → tasks cache invalidated {projectId, reason}
```

## Open Questions

> [!IMPORTANT]
> **Q1: Column Storage Strategy** — Should we use:
> - **(A)** `project_column_definitions` table only (full relational, audit-friendly)
> - **(B)** `projects.settings` JSONB field only (fast, simpler)
> - **(C)** Both — JSONB for fast client reads, table for source-of-truth

> I recommend **(A)** — dedicated table. It gives you full customization power, audit trail, and the ability to add constraints.
