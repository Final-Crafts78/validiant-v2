# Design System Strategy: The Architectural Verifier

## 1. Overview & Creative North Star
The Creative North Star for this design system is **"The Architectural Verifier."** 

In the world of field operations, data is often chaotic. This system is designed to act as a stabilizing force—transforming raw field verification into a high-end, editorial experience. We move beyond the "SaaS dashboard template" by employing a layout strategy rooted in **intentional asymmetry** and **tonal depth**. By utilizing a sophisticated interplay of deep navies and vibrant blues against a multi-tiered grayscale palette, we create an environment that feels authoritative yet breathable. The goal is to convey "Precision through Sophistication," ensuring that even the densest data tables feel like curated reports rather than cluttered spreadsheets.

---

## 2. Colors & Surface Logic
This system leverages a refined Material-style palette to move away from flat, uninspired interfaces.

### The "No-Line" Rule
**Explicit Instruction:** Designers are prohibited from using 1px solid borders to define sections or containers. Boundaries must be established through:
*   **Background Color Shifts:** Placing a `surface_container_low` element against a `surface` background.
*   **Tonal Transitions:** Using depth to signal the start of a new content area.

### Surface Hierarchy & Nesting
Treat the UI as a physical stack of premium materials. Use the `surface_container` tokens to create a "nested" logical flow:
*   **Base Layer (`surface`):** The canvas for the application.
*   **Structural Sections (`surface_container_low`):** Large regions like sidebars or secondary navigation.
*   **Actionable Content (`surface_container_lowest`):** Use the absolute white surface for primary data cards to make them "pop" against the subtle gray background.
*   **Active Overlays (`surface_bright`):** For temporary high-focus areas.

### The "Glass & Gradient" Rule
To inject "soul" into the professional interface:
*   **Glassmorphism:** For floating modals or top-tier navigation overlays, use `surface` colors at 80% opacity with a `20px` backdrop blur.
*   **Signature Textures:** Primary CTAs should not be flat. Use a subtle linear gradient (Top-Left to Bottom-Right) transitioning from `primary` (#0058be) to `primary_container` (#2170e4) to provide a tactile, "clickable" quality.

---

## 3. Typography
We utilize a dual-font system to balance editorial authority with high-density utility.

*   **Display & Headlines (Manrope):** Chosen for its geometric precision and modern "tech" feel. Use `display-lg` through `headline-sm` for page titles and high-level metrics. This font carries the "Editorial" weight of the system.
*   **Body & Labels (Inter):** The workhorse for data. Inter’s tall x-height and neutral character make it perfect for `body-md` and `label-sm` in dense verification tables.
*   **Hierarchy Note:** Always maintain a high contrast between headlines and body text. Use `on_surface_variant` for secondary labels to ensure the primary data (`on_surface`) remains the focal point.

---

## 4. Elevation & Depth
Depth is achieved through **Tonal Layering** rather than structural lines.

*   **The Layering Principle:** To create a card, do not draw a box. Instead, place a `surface_container_lowest` card on a `surface_container_low` background. This creates a soft, natural lift.
*   **Ambient Shadows:** For floating elements (e.g., dropdowns), use "Long Shadows": `box-shadow: 0 12px 40px rgba(15, 23, 42, 0.06);`. The shadow color must be a tinted version of `on_surface` to mimic natural light.
*   **The "Ghost Border" Fallback:** If a border is required for accessibility, use the `outline_variant` token at **15% opacity**. Never use 100% opaque borders.

---

## 5. Components

### Buttons
*   **Primary:** Gradient fill (`primary` to `primary_container`), `on_primary` text, `md` (0.375rem) corner radius.
*   **Secondary:** `secondary_container` fill with `on_secondary_container` text. No border.
*   **Tertiary:** Ghost style. No fill, `primary` text. Use only for low-emphasis actions.

### Cards & Data Lists
*   **Strict Rule:** Forbid the use of divider lines between list items. Use `8px` or `12px` of vertical white space to separate entries. 
*   **Verification Chips:** Use `secondary_fixed` for neutral states and `primary_fixed` for active/verified states. Keep the `full` (pill) radius for all chips to contrast against the `md` radius of cards.

### Input Fields
*   **Style:** Minimalist. Use `surface_container_highest` for the input background with a "Ghost Border" that transitions to a 2px `primary` bottom-border on focus.
*   **Validation:** Error states must use `error` text and an `error_container` subtle background highlight.

### Signature Component: The "Verification Stack"
A unique component for field ops: A vertical timeline of activity where "nodes" are connected not by lines, but by a subtle vertical gradient track using `surface_variant`, creating a sense of flow without visual clutter.

---

## 6. Do’s and Don’ts

### Do:
*   **Do** embrace negative space. If a table feels cramped, increase the padding, don't reduce the font size.
*   **Do** use `manrope` for numbers in dashboards to give them a high-end, "financial" aesthetic.
*   **Do** use asymmetrical layouts (e.g., a wide data column next to a narrow "Insights" column) to break the grid.

### Don't:
*   **Don't** use pure black (#000) for text. Always use `on_surface` (#191c1e).
*   **Don't** use standard 1px borders to separate table rows; use alternating `surface` and `surface_container_low` backgrounds (zebra striping) only if necessary; otherwise, use generous whitespace.
*   **Don't** use "Drop Shadows" on standard cards. Reserve elevation for elements that actually move or float above the layout.

# Design System Specification: The Obsidian Verifier

## 1. Overview & Creative North Star

This design system is a departure from the "generic SaaS" aesthetic. It is built upon the Creative North Star of **"The Obsidian Verifier."** Imagine a high-performance instrument carved from volcanic glass—dark, cold, and immensely powerful. It doesn't just display data; it validates it with clinical precision.

To move beyond the "template" look, this system rejects traditional structural boxing. We utilize **intentional asymmetry**, massive typographic contrast, and **tonal layering** to create a sense of three-dimensional space. The UI should feel like a series of illuminated data points floating within a deep, infinite void. We achieve "Premium" through restraint: use white space as a structural element and light as a functional tool.

---

## 2. Colors & Surface Logic

The palette is rooted in deep slates and electric blues, designed to pull the user's eye toward critical "verification" points.

### The "No-Line" Rule
Explicitly prohibit 1px solid, high-contrast borders for sectioning. Boundaries must be defined solely through background color shifts or subtle tonal transitions. Use `surface-container-low` sitting on `background` to define a region—not a stroke.

### Surface Hierarchy & Nesting
Treat the UI as a physical stack of frosted obsidian. 
- **Base Layer:** `background` (#0c1324)
- **Primary Layout Sections:** `surface-container-low` (#151b2d)
- **Nested Content/Cards:** `surface-container-highest` (#2e3447)
- **Interaction/Floating:** `surface-bright` (#33394c)

### The "Glass & Gradient" Rule
Floating elements (modals, dropdowns, floating nav) must utilize **Glassmorphism**. Apply `surface-variant` at 60% opacity with a `20px` backdrop-blur. 

**Signature Texture:** Main CTAs and Hero accents should use a linear gradient: `primary` (#adc6ff) to `primary-container` (#4d8eff) at a 135-degree angle. This provides a "glowing" soul that flat hex codes cannot replicate.

---

### Key Token Reference
*   **Primary (The Glow):** `#adc6ff` (Use for critical actions and data highlights)
*   **Background (The Void):** `#0c1324`
*   **Surface Lowest:** `#070d1f`
*   **Surface Highest:** `#2e3447`
*   **Tertiary (The Warning):** `#ffb786` (Use for high-energy alerts or specialized data streams)

---

## 3. Typography

The typographic voice is an interplay between the technical precision of **Inter** and the geometric authority of **Manrope**.

*   **Display & Headlines (Manrope):** These are your "Editorial" anchors. Use `display-lg` (3.5rem) with tight letter-spacing (-0.02em) to create an authoritative, "big-brand" feel. 
*   **Body & Labels (Inter):** Reserved for functional data. Inter's high x-height ensures readability against dark backgrounds. Use `body-md` (0.875rem) as the workhorse for all data entry.

**Hierarchy Tip:** To achieve an editorial look, skip levels. Place a `label-sm` (uppercase, tracked out +10%) directly above a `display-md` headline. The contrast in scale creates a sophisticated, "designed" appearance.

---

## 4. Elevation & Depth

We avoid the "card-on-gray" look of standard Material Design. Depth is achieved through **Tonal Layering**.

*   **The Layering Principle:** Place a `surface-container-lowest` card on a `surface-container-low` section. This creates a "recessed" or "carved" look, suggesting the UI is a singular, solid object rather than a collection of pasted boxes.
*   **Ambient Shadows:** For floating elements, use a 48px blur with 6% opacity. The shadow color should not be black; use a tinted version of `on-background` (#dce1fb) to simulate light refracting through the obsidian.
*   **The "Ghost Border" Fallback:** If a border is required for accessibility, use the `outline-variant` token at **15% opacity**. It should be a suggestion of an edge, not a hard line.

---

## 5. Components

### Buttons
*   **Primary:** Gradient fill (`primary` to `primary-container`). `ROUND_FOUR` (0.5rem) corners. White text (`on-primary`).
*   **Secondary:** Ghost style. `outline-variant` (20% opacity) border. 
*   **Tertiary:** No background, no border. `primary` text with a subtle underline on hover.

### Input Fields
*   **Surface:** `surface-container-highest`.
*   **Border:** None, except for a 2px bottom-stroke of `primary` when focused.
*   **State:** The field should "glow" subtly on focus using a `primary` outer-glow (4px blur, 10% opacity).

### Cards & Lists
*   **No Dividers:** Prohibit the use of line dividers between list items. Use 16px of vertical white space or alternate row backgrounds between `surface-container` and `surface-container-low`.
*   **Hover State:** On hover, a card should transition its background to `surface-bright` and shift 2px upward.

### Specialized Component: The Verifier Badge
A small, glassmorphic chip used to indicate data status. It uses a `tertiary-container` background at 20% opacity with a solid `tertiary` dot (4px) to the left of the label.

---

## 6. Do's and Don'ts

### Do
*   **DO** use extreme scale. A 12px label next to a 56px headline feels intentional and high-end.
*   **DO** use "Ink Traps." Use `surface-container-lowest` for inputs to make them feel carved into the interface.
*   **DO** leverage the "Obsidian" feel by keeping backgrounds dark and only using color for interactive elements.

### Don't
*   **DON'T** use pure #000000. It kills the depth. Use the `background` token (#0c1324).
*   **DON'T** use 100% opaque borders. They create "visual noise" and break the sleekness of the Obsidian Verifier.
*   **DON'T** use standard drop shadows. If a component feels flat, increase the tonal contrast between surfaces instead of adding a shadow.

- **Base Layer:** `background` (#0c1324)
- **Primary Layout Sections:** `surface-container-low` (#151b2d)
- **Nested Content/Cards:** `surface-container-highest` (#2e3447)
- **Interaction/Floating:** `surface-bright` (#33394c)

### The "Glass & Gradient" Rule
Floating elements (modals, dropdowns, floating nav) must utilize **Glassmorphism**. Apply `surface-variant` at 60% opacity with a `20px` backdrop-blur. 

**Signature Texture:** Main CTAs and Hero accents should use a linear gradient: `primary` (#adc6ff) to `primary-container` (#4d8eff) at a 135-degree angle. This provides a "glowing" soul that flat hex codes cannot replicate.

---

### Key Token Reference
*   **Primary (The Glow):** `#adc6ff` (Use for critical actions and data highlights)
*   **Background (The Void):** `#0c1324`
*   **Surface Lowest:** `#070d1f`
*   **Surface Highest:** `#2e3447`
*   **Tertiary (The Warning):** `#ffb786` (Use for high-energy alerts or specialized data streams)

---

## 3. Typography

The typographic voice is an interplay between the technical precision of **Inter** and the geometric authority of **Manrope**.

*   **Display & Headlines (Manrope):** These are your "Editorial" anchors. Use `display-lg` (3.5rem) with tight letter-spacing (-0.02em) to create an authoritative, "big-brand" feel. 
*   **Body & Labels (Inter):** Reserved for functional data. Inter's high x-height ensures readability against dark backgrounds. Use `body-md` (0.875rem) as the workhorse for all data entry.

**Hierarchy Tip:** To achieve an editorial look, skip levels. Place a `label-sm` (uppercase, tracked out +10%) directly above a `display-md` headline. The contrast in scale creates a sophisticated, "designed" appearance.

---

## 4. Elevation & Depth

We avoid the "card-on-gray" look of standard Material Design. Depth is achieved through **Tonal Layering**.

*   **The Layering Principle:** Place a `surface-container-lowest` card on a `surface-container-low` section. This creates a "recessed" or "carved" look, suggesting the UI is a singular, solid object rather than a collection of pasted boxes.
*   **Ambient Shadows:** For floating elements, use a 48px blur with 6% opacity. The shadow color should not be black; use a tinted version of `on-background` (#dce1fb) to simulate light refracting through the obsidian.
*   **The "Ghost Border" Fallback:** If a border is required for accessibility, use the `outline-variant` token at **15% opacity**. It should be a suggestion of an edge, not a hard line.

---

## 5. Components

### Buttons
*   **Primary:** Gradient fill (`primary` to `primary-container`). `ROUND_FOUR` (0.5rem) corners. White text (`on-primary`).
*   **Secondary:** Ghost style. `outline-variant` (20% opacity) border. 
*   **Tertiary:** No background, no border. `primary` text with a subtle underline on hover.

### Input Fields
*   **Surface:** `surface-container-highest`.
*   **Border:** None, except for a 2px bottom-stroke of `primary` when focused.
*   **State:** The field should "glow" subtly on focus using a `primary` outer-glow (4px blur, 10% opacity).

### Cards & Lists
*   **No Dividers:** Prohibit the use of line dividers between list items. Use 16px of vertical white space or alternate row backgrounds between `surface-container` and `surface-container-low`.
*   **Hover State:** On hover, a card should transition its background to `surface-bright` and shift 2px upward.

### Specialized Component: The Verifier Badge
A small, glassmorphic chip used to indicate data status. It uses a `tertiary-container` background at 20% opacity with a solid `tertiary` dot (4px) to the left of the label.

---

## 6. Do's and Don'ts

### Do
*   **DO** use extreme scale. A 12px label next to a 56px headline feels intentional and high-end.
*   **DO** use "Ink Traps." Use `surface-container-lowest` for inputs to make them feel carved into the interface.
*   **DO** leverage the "Obsidian" feel by keeping backgrounds dark and only using color for interactive elements.

### Don't
*   **DON'T** use pure #000000. It kills the depth. Use the `background` token (#0c1324).
*   **DON'T** use 100% opaque borders. They create "visual noise" and break the sleekness of the Obsidian Verifier.
*   **DON'T** use standard drop shadows. If a component feels flat, increase the tonal contrast between surfaces instead of adding a shadow.



# Design System Strategy: The Architectural Verifier

## 1. Overview & Creative North Star
The Creative North Star for this design system is **"The Architectural Verifier."** 

In the world of field operations, data is often chaotic. This system is designed to act as a stabilizing force—transforming raw field verification into a high-end, editorial experience. We move beyond the "SaaS dashboard template" by employing a layout strategy rooted in **intentional asymmetry** and **tonal depth**. By utilizing a sophisticated interplay of deep navies and vibrant blues against a multi-tiered grayscale palette, we create an environment that feels authoritative yet breathable. The goal is to convey "Precision through Sophistication," ensuring that even the densest data tables feel like curated reports rather than cluttered spreadsheets.

---

## 2. Colors & Surface Logic
This system leverages a refined Material-style palette to move away from flat, uninspired interfaces.

### The "No-Line" Rule
**Explicit Instruction:** Designers are prohibited from using 1px solid borders to define sections or containers. Boundaries must be established through:
*   **Background Color Shifts:** Placing a `surface_container_low` element against a `surface` background.
*   **Tonal Transitions:** Using depth to signal the start of a new content area.

### Surface Hierarchy & Nesting
Treat the UI as a physical stack of premium materials. Use the `surface_container` tokens to create a "nested" logical flow:
*   **Base Layer (`surface`):** The canvas for the application.
*   **Structural Sections (`surface_container_low`):** Large regions like sidebars or secondary navigation.
*   **Actionable Content (`surface_container_lowest`):** Use the absolute white surface for primary data cards to make them "pop" against the subtle gray background.
*   **Active Overlays (`surface_bright`):** For temporary high-focus areas.

### The "Glass & Gradient" Rule
To inject "soul" into the professional interface:
*   **Glassmorphism:** For floating modals or top-tier navigation overlays, use `surface` colors at 80% opacity with a `20px` backdrop blur.
*   **Signature Textures:** Primary CTAs should not be flat. Use a subtle linear gradient (Top-Left to Bottom-Right) transitioning from `primary` (#0058be) to `primary_container` (#2170e4) to provide a tactile, "clickable" quality.

---

## 3. Typography
We utilize a dual-font system to balance editorial authority with high-density utility.

*   **Display & Headlines (Manrope):** Chosen for its geometric precision and modern "tech" feel. Use `display-lg` through `headline-sm` for page titles and high-level metrics. This font carries the "Editorial" weight of the system.
*   **Body & Labels (Inter):** The workhorse for data. Inter’s tall x-height and neutral character make it perfect for `body-md` and `label-sm` in dense verification tables.
*   **Hierarchy Note:** Always maintain a high contrast between headlines and body text. Use `on_surface_variant` for secondary labels to ensure the primary data (`on_surface`) remains the focal point.

---

## 4. Elevation & Depth
Depth is achieved through **Tonal Layering** rather than structural lines.

*   **The Layering Principle:** To create a card, do not draw a box. Instead, place a `surface_container_lowest` card on a `surface_container_low` background. This creates a soft, natural lift.
*   **Ambient Shadows:** For floating elements (e.g., dropdowns), use "Long Shadows": `box-shadow: 0 12px 40px rgba(15, 23, 42, 0.06);`. The shadow color must be a tinted version of `on_surface` to mimic natural light.
*   **The "Ghost Border" Fallback:** If a border is required for accessibility, use the `outline_variant` token at **15% opacity**. Never use 100% opaque borders.

---

## 5. Components

### Buttons
*   **Primary:** Gradient fill (`primary` to `primary_container`), `on_primary` text, `md` (0.375rem) corner radius.
*   **Secondary:** `secondary_container` fill with `on_secondary_container` text. No border.
*   **Tertiary:** Ghost style. No fill, `primary` text. Use only for low-emphasis actions.

### Cards & Data Lists
*   **Strict Rule:** Forbid the use of divider lines between list items. Use `8px` or `12px` of vertical white space to separate entries. 
*   **Verification Chips:** Use `secondary_fixed` for neutral states and `primary_fixed` for active/verified states. Keep the `full` (pill) radius for all chips to contrast against the `md` radius of cards.

### Input Fields
*   **Style:** Minimalist. Use `surface_container_highest` for the input background with a "Ghost Border" that transitions to a 2px `primary` bottom-border on focus.
*   **Validation:** Error states must use `error` text and an `error_container` subtle background highlight.

### Signature Component: The "Verification Stack"
A unique component for field ops: A vertical timeline of activity where "nodes" are connected not by lines, but by a subtle vertical gradient track using `surface_variant`, creating a sense of flow without visual clutter.

---

## 6. Do’s and Don’ts

### Do:
*   **Do** embrace negative space. If a table feels cramped, increase the padding, don't reduce the font size.
*   **Do** use `manrope` for numbers in dashboards to give them a high-end, "financial" aesthetic.
*   **Do** use asymmetrical layouts (e.g., a wide data column next to a narrow "Insights" column) to break the grid.

### Don't:
*   **Don't** use pure black (#000) for text. Always use `on_surface` (#191c1e).
*   **Don't** use standard 1px borders to separate table rows; use alternating `surface` and `surface_container_low` backgrounds (zebra striping) only if necessary; otherwise, use generous whitespace.
*   **Don't** use "Drop Shadows" on standard cards. Reserve elevation for elements that actually move or float above the layout.
