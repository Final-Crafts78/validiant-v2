## Executive Summary    

Validiant-v2 is a Turborepo monorepo with five workspaces:

| Workspace | Path | Technology | Purpose |
|-----------|------|------------|---------|
| `apps/web` | Next.js 14 (App Router) | Frontend dashboard | `www.validiant.in` |
| `apps/api` | Hono on Cloudflare Workers | REST API | `api.validiant.in` |
| `apps/mobile` | React Native (Expo) | Field agent app | Play Store / App Store |
| `packages/shared` | TypeScript library | Types, schemas, utils | Shared across all apps |
| `packages/ui` | React component library | Reusable UI primitives | Shared across web + mobile |
| `packages/config` | Config files | ESLint, TS config | Shared config |

**Core insight**: The Projects system must evolve from a fixed-schema task container into a
**configurable data universe** where each project defines its own Types, Columns, and Records.
Everything else (field portals, client portals, inbound APIs, analytics) cascades from that.

**What the Existing `verification_types` + `case_field_values` System Already Does:**

This is critical. The codebase already HAS a proto-schema engine:
- `verification_types` defines a named type with a `field_schema` JSONB array of `FieldSchema` objects
- Each `FieldSchema` has: `fieldKey`, `label`, `type` (text/textarea/boolean/photo-request/signature/pdf-upload/number)
- `case_field_values` stores actual data in an EAV pattern with typed columns: `value_text`, `value_number`, `value_date`, `value_boolean`, `value_json`
- `case_document_uploads` handles file uploads with integrity hashing and geo-tagging

**This means the new Types/Columns/Records system should EVOLVE from this existing pattern, not replace it from scratch.** The `verification_types` → `project_types` migration is a rename + enhancement, not a greenfield build.

### 1.2 Key Corrections to the Original Plan

| Original Claim | Reality | Impact on Plan |
|----------------|---------|----------------|
| "project.types.ts defines progress field" | ✅ Correct — `progress: number` on `Project` interface | None |
| "project has autoDispatchVerified, themeColor, logoUrl" | ✅ Exists on the `projects` DB table, but NOT on the shared `Project` interface — the DB schema has them, the TS type doesn't expose all | Schema tab's project settings need to read from DB fields, not just the TS type |
| "notification.types.ts already exists" | ✅ Correct — 21 notification types, 4 channels | Can extend, not create from scratch |
| "customFields: CustomFieldValue[]" is an afterthought | ✅ Correct — `customFields` is `JSONB` on tasks table | The EAV `case_field_values` table is the real field engine already |
| "No SLA engine" | ⚠️ Partially wrong — tasks already have `sla_at_risk_notified` and `sla_breached_notified` columns, and `verification_types` has `sla_override_hours` | Extend existing SLA columns, don't build from zero |
| "Sub-accounts stored in Supabase auth.users" | ❌ Wrong — Validiant uses Neon PostgreSQL via Drizzle, not Supabase. Auth is custom JWT-based (see `auth.service.ts`, `auth.controller.ts`) | Sub-accounts stored in `users` table with metadata, or a dedicated `org_sub_accounts` table |
| "Supabase Realtime subscriptions" | ❌ Wrong — Validiant uses Cloudflare Durable Objects for real-time (see `wrangler.toml` line 26-28, `RealtimeRoom` class) | Real-time via Durable Objects WebSocket, not Supabase channels |
| "Supabase Storage for file upload" | ❌ Wrong — Validiant uses Cloudflare R2 (see `wrangler.toml` BACKUP_BUCKET binding, `storage.service.ts`) | File uploads via R2 or existing upload flow |
| "SQL CREATE TABLE" syntax in plan | ❌ Misleading — schema changes are Drizzle ORM table definitions in `schema.ts`, not raw SQL | All schema changes are Drizzle `pgTable()` definitions |
| "POST https://api.validiant.io/v1/ingest/{projectKey}" | ⚠️ Close — actual domain is `api.validiant.in` (see `wrangler.toml` route) | Use `api.validiant.in` not `api.validiant.io` |
| "portal.validiant.io" for client portal | ❌ No such domain exists | Client portal will be a route under the main web app or a subdomain to be configured |

## Part 2 — The Schema Engine Architecture

### 2.1 The Data Hierarchy (Corrected)

```
Organization (existing: `organizations` table)
└── Project (existing: `projects` table)
    ├── Project API Key (existing: `client_api_keys` table — rename/enhance to `project_api_keys`)
    ├── Types  ← NEW (evolve from `verification_types`)
    │   ├── Type: "Identity Verification"
    │   │   └── Columns: [Applicant Name, Aadhaar, DOB, ...]  ← NEW (evolve from `field_schema` JSONB)
    │   ├── Type: "Address Check"
    │   │   └── Columns: [Resident Name, Address, ...]
    │   └── Type: "Employment Check"
    │       └── Columns: [Company Name, HR Contact, ...]
    ├── Records  ← NEW (evolve from `tasks` + `case_field_values`)
    ├── Members  (existing: `project_members` table)
    └── Settings (existing: `projects.settings` JSONB)
```

### 2.2 Migration Strategy: Evolve, Don't Replace

**Phase approach to avoid breaking changes:**

1. **Create NEW tables** (`project_types`, `type_columns`, `records`, `record_history`) alongside existing tables
2. **Keep existing `tasks` and `verification_types` tables** working during transition
3. **Data migration script**: Copy `verification_types` → `project_types`, `field_schema` fields → `type_columns` rows
4. **Dual-mode support**: Projects can use either Tasks (legacy) or Records (new) — controlled by a `dataMode: 'tasks' | 'records' | 'hybrid'` field in `projects.settings`
5. **Deprecate** Tasks-only mode after all orgs have migrated

export const projectTypes = pgTable('project_types', {
  id: uuid('id').primaryKey().defaultRandom(),
  projectId: uuid('project_id').notNull()
    .references(() => projects.id, { onDelete: 'cascade' }),
  name: text('name').notNull(),
  description: text('description'),
  icon: text('icon'),          // Emoji or Lucide icon key
  color: text('color'),
  order: integer('order').default(0),
  settings: jsonb('settings').default({}),
  // Settings schema: { requiredFields: string[], slaHours?: number,
  //   gpsRequired: boolean, allowedCreators: ('admin'|'member'|'field_agent')[] }
  createdBy: uuid('created_by').references(() => users.id),
  createdAt: timestamp('created_at', { mode: 'date', withTimezone: true }).notNull().defaultNow(),
  updatedAt: timestamp('updated_at', { mode: 'date', withTimezone: true }).default(sql`CURRENT_TIMESTAMP`).notNull(),
}, (table) => ({
  projectIdx: index('project_types_project_id_idx').on(table.projectId),
  projectNameUnique: unique('project_types_project_name_unique').on(table.projectId, table.name),
}));

  createdVia: 'web' | 'mobile' | 'api' | 'csv_import';
  clientId?: string;
  gpsLat?: number;
  gpsLng?: number;
  gpsAccuracy?: number;
  submittedAt?: Date;
  closedAt?: Date;
  createdAt: Date;
  updatedAt: Date;
}

### 3.3 Auth Flow for Sub-Accounts

The authentication flow routes through the existing `apps/api/src/services/auth.service.ts` and `apps/api/src/controllers/auth.controller.ts`.

**Field Agent invite flow:**
1. Admin calls `POST /api/v1/orgs/:orgId/sub-accounts` with `{ accountType: 'field_agent', name, phone, email, projectAccess }`
2. Backend creates a `users` record with `role: 'user'` + an `org_sub_accounts` record with `accountType: 'field_agent'`
3. Backend sends activation link via email (existing `email.service.ts`)
4. Agent clicks link → sets password → auto-logged in
5. Middleware checks `org_sub_accounts` on login → routes to field portal if `accountType === 'field_agent'`

**Client Viewer flow:**
1. Admin calls `POST /api/v1/orgs/:orgId/sub-accounts` with `{ accountType: 'client_viewer', clientId, name, email, projectAccess }`
2. Backend generates a unique portal token (stored hashed in `org_sub_accounts` metadata)
3. Client accesses `https://www.validiant.in/portal/client/:orgSlug/:token` ← **path-based, NOT a subdomain**
4. Token maps to `org_sub_accounts` → renders read-only record view filtered by `clientId`

> **Decision: Path-based portal routing (locked in)**
>
> The client portal is a path under the existing Vercel-hosted web app:
> `https://www.validiant.in/portal/client/:orgSlug/:token`
>
> Rationale: The web app is deployed on Vercel. A subdomain (e.g. `portal.validiant.in`)
> would require additional DNS CNAME records, a separate Vercel project or rewrite rules,
> and complicates the deployment pipeline. A path-based approach (`/portal/...`) works
> with the existing Next.js App Router — just add pages under `apps/web/src/app/portal/`.
> The field agent portal follows the same pattern: `/portal/field/...`.
>
> If a dedicated subdomain is needed later (e.g. for white-labeling), it can be added
> via Vercel's domain aliases feature without changing the route structure.

### 5.8 Ingest API Rate Limiting (Critical — must ship with the endpoint)

`POST /api/v1/ingest/:projectKey` is a public endpoint authenticated only by an API key,
not by JWT session. This makes it the highest-abuse-risk surface in the platform. Rate
limiting **must** be implemented before the endpoint goes live, not after.

**Two-layer defense:**

#### Layer 1: Cloudflare Rate Limiting Rules (infrastructure-level)

Applied *before* the Worker executes — blocks abusive IPs at the edge with zero CPU cost.

**Add to Cloudflare Dashboard** (or via Terraform/Pulumi if infra-as-code):

| Rule | Matching Expression | Threshold | Action | Window |
|------|--------------------|-----------|----|--------|
| Ingest IP Limit | `http.request.uri.path matches "^/api/v1/ingest/"` | 100 requests | Challenge | 60 seconds |
| Ingest IP Hard Limit | `http.request.uri.path matches "^/api/v1/ingest/"` | 500 requests | Block | 60 seconds |

#### Layer 2: Per-API-Key Rate Limiting (application-level)

Applied inside the Worker using the existing Upstash Redis integration (already configured
in `wrangler.toml` secrets via `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN`).

**Implementation in `ingest.routes.ts` middleware:**

```typescript
import { rateLimit } from '../middleware/rateLimit';

// Apply per-key rate limit: 60 requests per 60 seconds per API key
// Uses the existing rateLimit middleware pattern from apps/api/src/middleware/rateLimit.ts
// but with the API key hash as the identifier instead of IP/userId

ingestRouter.use('/:projectKey', async (c, next) => {
  const apiKeyHeader = c.req.header('Authorization')?.replace('Bearer ', '');
  if (!apiKeyHeader) {
    return c.json({ error: 'Missing API key' }, 401);
  }
  // Hash the key for the rate limit bucket (don't use raw key as Redis key)
  const keyHash = await crypto.subtle.digest(
    'SHA-256',
    new TextEncoder().encode(apiKeyHeader)
  ).then(buf =>
    Array.from(new Uint8Array(buf))
      .map(b => b.toString(16).padStart(2, '0'))
      .join('')
      .slice(0, 16)
  );
  
  return rateLimit(60, 60, `rl:ingest:${keyHash}`)(c, next);
});
```

**Rate limit tiers by subscription plan** (Phase 6 enhancement):

| Plan | Requests/minute | Requests/day | Burst |
|------|----------------|-------------|-------|
| Free | 10 | 500 | 20 |
| Starter | 60 | 10,000 | 100 |
| Pro | 300 | 100,000 | 500 |
| Enterprise | Custom | Custom | Custom |

**Response headers** (follow standard `RateLimit-*` headers):
```
RateLimit-Limit: 60
RateLimit-Remaining: 42
RateLimit-Reset: 1717500000
Retry-After: 18   (only on 429 responses)
```

**429 response body:**
```json
{
  "error": "rate_limit_exceeded",
  "message": "Too many requests. Limit: 60/minute per API key.",
  "retryAfter": 18
}
```

**Server-side validation flow (in `ingest.service.ts`):**
1. Resolve `typeId` — if `type` (name) is given, look up `project_types` by `name + projectId`

8. Fire automations matching trigger `RECORD_CREATED_VIA_API`
9. Return `{ recordId, displayId, status: 'accepted' }` — target: < 100ms response

```typescript
/**
 * One-time data migration: verification_types → project_types + type_columns
 *
 * For each verification_type linked to an org:
 *   1. Create a project_type row (name, settings, org context)
 *   2. For each entry in field_schema JSONB array → create a type_column row
 *   3. Mapping: fieldKey → column.key, type → column.columnType, required → settings.required
 *
 * Run: npx tsx apps/api/src/db/migrate-verification-to-types.ts
 */
import { db } from './index';
import { verificationTypes, projectTypes, typeColumns } from './schema';
import { eq, isNotNull } from 'drizzle-orm';

const FIELD_TYPE_MAP: Record<string, string> = {
  'text':           'text',
  'textarea':       'long_text',
  'boolean':        'checkbox',
  'photo-request':  'photo_capture',
  'signature':      'signature',
  'pdf-upload':     'file_upload',
  'number':         'number',
};

async function migrateVerificationTypes() {
  console.log('Starting verification_types → project_types migration...');
  
  // 1. Fetch all org-scoped verification types (skip system templates)
  const orgTypes = await db
    .select()
    .from(verificationTypes)
    .where(isNotNull(verificationTypes.organizationId));
  
  console.log(`Found ${orgTypes.length} org-scoped verification types to migrate.`);

  for (const vt of orgTypes) {
    // 2. Find project(s) linked to this org.
    //    Since verification_types are org-scoped but project_types are project-scoped,
    //    we need to associate each type with a target project.
    //    Strategy: create type in ALL active projects for this org,
    //    or create in a designated "default" project.
    //    This decision should be confirmed with the user before running.
    
    // For safety, create a project_type per verification_type per project:
    // (In practice, filter to projects that actually use this verification type)
    const projectsForOrg = await db.query.projects.findMany({
      where: (p, { eq: _eq }) => _eq(p.organizationId, vt.organizationId!),
    });

    for (const project of projectsForOrg) {
      // 3. Insert project_type
      const [newType] = await db.insert(projectTypes).values({
        projectId: project.id,
        name: vt.name,
        description: `Migrated from verification type: ${vt.code}`,
        settings: {
          slaHours: vt.slaOverrideHours ?? undefined,
          gpsRequired: false,
        },
      }).returning();

      console.log(`  Created type "${vt.name}" in project "${project.name}" → ${newType.id}`);

      // 4. Convert field_schema array → type_columns rows
      const fields = (vt.fieldSchema as any[]) || [];
      for (let i = 0; i < fields.length; i++) {
        const f = fields[i];
        const mappedType = FIELD_TYPE_MAP[f.type] || 'text';

        await db.insert(typeColumns).values({
          typeId: newType.id,
          projectId: project.id,
          name: f.label || f.fieldKey,
          key: f.fieldKey,
          columnType: mappedType,
          options: f.options ? { choices: f.options } : undefined,
          settings: {
            required: f.required ?? false,
            showInList: true,
            showInCard: true,
            showInMobile: true,
            hint: f.prompt || undefined,
            visibleTo: f.readRoles?.map((r: string) =>
              r === 'executive' ? 'member' : r === 'client' ? 'client_portal' : r
            ) || ['admin', 'member'],
          },
          order: i,
        });
      }
      console.log(`  Migrated ${fields.length} columns for type "${vt.name}"`);
    }
  }

  console.log('✅ Migration complete.');
}

if (require.main === module) {
  migrateVerificationTypes()
    .then(() => process.exit(0))
    .catch((err) => { console.error('Migration failed:', err); process.exit(1); });
}

export { migrateVerificationTypes };
```

## Part 8 — Key Technical Decisions

### 8.1 Records vs Tasks Co-existence

Projects should support BOTH Tasks (the existing system) and Records (the new system) simultaneously during the transition. This is controlled by a `dataMode` field in `projects.settings` JSONB:

```typescript
interface ProjectSettings {
  // ... existing fields ...
  dataMode?: 'tasks' | 'records' | 'hybrid'; // Default: 'tasks' for existing projects
}
```

- `tasks` mode: Project uses the existing DataExplorerTab with tasks
- `records` mode: Project uses the new RecordsTab with Types/Columns
- `hybrid` mode: Both tabs are visible (for migration period)

New projects created after Phase 1 deployment default to `records` mode.

### 8.2 Record Numbering Strategy (Concurrency-Safe)

Record numbers must be sequential per project (e.g. BGV-0001, BGV-0002, ...) and **must never
produce duplicates** even under concurrent inserts from the web UI, field portal, and inbound
API simultaneously.

**Implementation in `records.service.ts`:**

```typescript
import { db } from '../db';
import { records } from '../db/schema';
import { eq, sql, max } from 'drizzle-orm';

async function getNextRecordNumber(projectId: string): Promise<number> {
  // Use a Postgres transaction with advisory lock scoped to the project.
  // pg_advisory_xact_lock takes a bigint key — we hash the project UUID to i64.
  return await db.transaction(async (tx) => {
    // 1. Acquire an advisory lock scoped to this project.
    //    hashtext() returns a stable int4; cast to bigint for pg_advisory_xact_lock.
    //    The lock is automatically released when the transaction commits/rolls back.
    await tx.execute(
      sql`SELECT pg_advisory_xact_lock(hashtext(${projectId}::text)::bigint)`
    );

    // 2. Get the current max number within the lock.
    const [result] = await tx
      .select({ maxNum: max(records.number) })
      .from(records)
      .where(eq(records.projectId, projectId));

    const nextNumber = (result?.maxNum ?? 0) + 1;

    return nextNumber;
  });
}
```

**Why advisory lock and not `FOR UPDATE`?**
- There is no dedicated "sequence" row to lock. `SELECT MAX(...) FOR UPDATE` would lock
  ALL matching rows in the `records` table for that project, causing severe contention.
- `pg_advisory_xact_lock` is a lightweight mutex that only blocks other callers asking
  for the same lock key. It does NOT block reads or unrelated writes.
- The lock is transaction-scoped: it auto-releases on commit/rollback, so a crashed worker
  cannot hold it forever.

**Why not a Postgres SERIAL/SEQUENCE?**
- SERIAL is global, not per-project. We need numbering scoped to each project.
- A separate `sequences` table is an option but adds complexity without benefit over
  advisory locks.

**Display ID construction:**
```typescript
const displayId = `${project.key}-${nextNumber.toString().padStart(4, '0')}`;
// e.g. "BGV-0042", "DLV-0001"
```

The `displayId` is stored on the record row for fast lookups (no joins needed).

### 8.3 Data Storage: JSONB for Record Data

Record data is stored in a single `data JSONB` column (not EAV). This is different from the
existing `case_field_values` EAV approach.

**Comparison table:**

| Factor | JSONB (`records.data`) | EAV (`case_field_values`) |
|--------|------------------------|---------------------------|
| Read performance | ✅ Single row read | ❌ N rows per record |
| Write performance | ✅ Single INSERT | ❌ N INSERTs per record |
| Query flexibility | ⚠️ JSONB operators required | ✅ Standard SQL WHERE |
| Schema validation | ⚠️ App-level only | ⚠️ App-level only |
| Index support | ✅ GIN indexes on JSONB | ✅ B-tree on field_key + value |
| Cold start latency | ✅ 1 round-trip | ❌ N+1 round-trips |

**Why JSONB wins for Cloudflare Workers + Neon Postgres specifically:**

1. **Connection cost.** Cloudflare Workers connect to Neon over a WebSocket driver
   (`@neondatabase/serverless`). Each SQL query incurs a round-trip across the internet.
   EAV requires 1 query to fetch the record + 1 query to fetch N field rows (or a JOIN
   that returns N result rows). JSONB is always 1 row returned per record read, which
   is dramatically faster on a serverless connection pool.

2. **Worker CPU limits.** Workers have a 30ms CPU time budget (paid plan) per request.
   Parsing 1 JSON object from a single row is significantly cheaper than materializing
   and mapping N EAV rows into an object. For records with 20+ columns, this difference
   is meaningful.

3. **GIN indexing.** Neon Postgres fully supports GIN indexes on JSONB. For the queries
   we need ("find all records where `data->>'status' = 'verified'`"), a GIN index with
   `jsonb_path_ops` provides efficient lookup without needing a separate index table.
   Example migration:
   ```sql
   CREATE INDEX records_data_gin ON records USING GIN (data jsonb_path_ops);
   ```

4. **Backward compatibility.** The existing EAV table (`case_field_values`) remains
   untouched for legacy `tasks` with `verification_type_id` references. No data is lost.
   The two patterns coexist cleanly — JSONB for new Records, EAV for old Tasks.

5. **Eventual filtering strategy.** If cross-column filtering becomes a bottleneck,
   we can add Postgres generated columns for hot fields:
   ```sql
   ALTER TABLE records ADD COLUMN status_val TEXT GENERATED ALWAYS AS (data->>'status') STORED;
   CREATE INDEX records_status_val ON records (project_id, status_val);
   ```
   This gives us B-tree speed on specific columns without abandoning the JSONB model.

### 8.4 Real-time Architecture

Use the existing Cloudflare Durable Objects pattern:
- Each project has a Durable Object room: `project:{projectId}`.
- When a record is created/updated, the service broadcasts to the room.
- Frontend subscribes via WebSocket through the existing real-time hook.
- **Why**: Avoids extra dependencies (like Supabase or Ably) while stays within the Cloudflare Workers free tier.

### 8.8 Schema Evolution & Key Immutability

Data integrity is paramount. In a JSONB model, changing a column key breaks existing records.

- **Decision**: The `key` of a `TypeColumn` is **MUTABLE ONLY VIA MIGRATION**.
  - In the UI, the `key` field is disabled after creation.
  - Users can change the `label` (Display Name) anytime.
  - To change a `key`, the API will provide a "Refactor Key" utility that runs a batch update on the `records` table for that project:
    ```sql
    UPDATE records SET data = (data - 'old_key') || jsonb_build_object('new_key', data->'old_key')
    WHERE project_id = :projectId AND type_id = :typeId;
    ```
- **Type Safety**: When a column type changes (e.g., `NUMBER` to `TEXT`), the UI will warn that existing data might be invalidated by frontend validators.

### 8.9 Advanced Search & GIN Index strategy

To support fast searching across millions of records within the **Neon Free Tier (500MB storage)**, we utilize specialized indexes.

- **Primary GIN Index**:
  ```sql
  CREATE INDEX records_data_path_idx ON records USING GIN (data jsonb_path_ops);
  ```
  *Why:* `jsonb_path_ops` indexes are smaller and faster than default GIN indexes, fitting perfectly within free tier storage limits.
- **Project-Scoped GIN Indexing**:
  Since 99% of queries are scoped to a `project_id`, we use a composite approach or filtered indexes to keep index sizes minimal.
- **Search Logic**: Global search (Cmd+K) uses the `@neondatabase/serverless` driver to run:
  ```sql
  SELECT * FROM records WHERE data @> jsonb_build_object('search_field', 'query_val');
  ```

### 8.10 Record Projection & Field-Level Security (RBAC)

We do NOT send the raw `data` JSONB object to the frontend. This prevents leaking "Internal Only" fields.

- **Projection Pipeline**:
  1. Service fetches `Record` + `TypeColumns`.
  2. Filter `TypeColumns` by user's current role (`settings.visibleTo`).
  3. Construct a new `data` object containing ONLY the keys the user is authorized to see.
  4. Return the projected object.
- **System Fields**: Fields starting with `_` (e.g., `_metadata`, `_version`) are reserved for system use and are stripped from all public API responses unless explicitly requested by an Admin.
